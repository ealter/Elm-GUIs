Menus in Elm

By Eliot Alter and Max Goldstein

For our final project in Comp150-FP, we co-wrote a paper on our attempt to create desktop-style menus in Elm. It outlines the challenges posed and the solutions we found. Primarily, we were able to detect hover information about the elements in a Signal Element without modifying the compiler or runtime, but in a way that is very non-obvious (hopefully). It is a workaround for the lack of signals of signals. Any sufficiently advanced GUI using Graphics.Input will be affected by the problems we outline. We also include an overview of Elm targeted for those with a background in FP, with particular attention paid to explaining why signals of signals are disallowed. We also briefly analyze TodoFRP as the current state-of-the-art in Elm GUIs, and the HTML widget wrappers in Graphics.Input. You can read our paper here.

We also provided several demos highlighting various functionality of our menu system. The source code for each example can be found in our GitHub repository.

  • main.elm is a standard menus with a few nested directories.
  • dynamic.elm shows off a menu with non-constant strings. Certain parts of the menu change on keyboard strokes and mouse clicks.
  • indexes.elm looks just like main.elm except it tests the stringsAgain and treeAtPath functions.
  • maybeBlank.elm tests how the code handles blank strings. It demonstrates a bug in the hoverable detection of Elm. Put your mouse in the middle of the first menu item and wait for the menu to change. After the menu changes, the mouse stops hovering over the first item, but the first submenu remains expanded.
  • clicks.elm demonstrates click detection.
